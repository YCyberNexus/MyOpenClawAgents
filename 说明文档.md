> **归档说明**：本文记录的是 `SKILL_VERSION=2026-04-25.2` 时代的旧架构，不再作为当前运行合同。当前调度、spawn、路径和状态文件规范以 `CLAUDE.md`、`workspace-acpx_auto_tester/AGENTS.md`、`workspace-acpx_auto_tester/SOUL.md` 和 `workspace-acpx_auto_tester/skills/gitlab_issue_campaign_dispatcher/SKILL.md` 为准；当前子任务创建必须使用匿名 `sessions_spawn` + `RUN_CHILD_COMPLETION_CALLBACK`，不得使用命名 session 或同步等待子 agent 终态。

> 当前版本：`SKILL_VERSION = 2026-04-25.2`
> 当前分支：`dev_per-attempt-symlink`
> 适用范围：基于 OpenClaw + acpx + Claude Code 的 GitLab 自动 issue 处理 agent

## 1. 目标与定位

`UiAutoTester` 是一个**非交互、长周期调度运行**的 GitLab issue 自动化 agent。运维只配一次，调度器按周期反复触发，agent 把 GitLab 项目里指定 IID 范围内的 issue 逐条处理：

1. 拉取 issue → 转 label `doing`
2. 用 `acpx` 调用 Claude Code 在干净 worktree 里干活（按 issue 描述生成测试 spec / 执行报告）
3. 提交 → 推到 issue 专属分支 → 创建一条向 `master` 的 MR（不自动 merge）
4. 把 issue label 切到 `done`，把这次执行的摘要回写到 issue 评论
5. 失败时按"可重试 / 不可恢复"分类落盘，不会自爆

reviewer 在 GitLab 上验收。如果发现 Claude 没真跑完，把 label 从 `done` 改成 `continue` 并在 issue 上加一条 comment 说明补救步骤，下一轮 agent 自动用 continue 模式在原分支续跑。

## 2. 双角色执行模型

agent 工作空间 `workspace-acpx_auto_tester/skills/` 下有两个 SKILL，分工明确：

```
┌────────────────────────────────────────────────────────────────────┐
│  Dispatcher Session (固定，每个 scheduler tick 唤醒同一个)         │
│  ─────────────────────────────────────────────────                │
│  - 加载 / 校验 campaign 状态                                       │
│  - 调 GitLab 校验真实 label（reconciliation）                      │
│  - 串行选下一个 IID，spawn 一个 dedicated session                  │
│  - 收到 executor 终态后清 active_issue_iid，循环                    │
└────────────────────────────────────────────────────────────────────┘
                            │ sessions_spawn (串行、阻塞、单调用)
                            ▼
┌────────────────────────────────────────────────────────────────────┐
│  Executor Session (一次性，名为 issue-<project>-<iid>)             │
│  ─────────────────────────────────────────────────                │
│  - 拉 / fetch 主仓库                                                │
│  - git worktree 切出 attempt-NNN 的 working tree                   │
│  - symlink hulat_dir 到 worktree/_hulat                             │
│  - 构建 prompt（含历次 attempt summary + reviewer 评论）           │
│  - acpx claude exec -f 同步运行                                     │
│  - leak guard → commit → 强制推送 → MR (复用) → 标 done            │
│  - 把摘要回 post 到 issue 评论                                      │
└────────────────────────────────────────────────────────────────────┘
```

两个 session 通过磁盘状态文件交接：dispatcher 写 spawn 触发，executor 跑完写 per-issue / per-attempt state，dispatcher 再 read。

## 3. 磁盘布局

```
/data/${PROJECT}/                                    ← 主 git 仓库（hosts worktrees）
/data/openclaw_work/${PROJECT}/                      ← 所有 agent 产物，仓库外
    openclaw_state/
        campaign_state.json                          ← campaign 级缓存
        campaign.lock                                ← flock target
    openclaw_log/
        dispatcher/
            reconcile-<ts>.json                      ← reconciliation 证据文件
    issues/
        issue-<iid>/                                 ← 单 issue 全部产物
            state.json                               ← 跨 attempt 的 issue 状态
            attempts/
                attempt-001/
                    worktree/                        ← Claude Code 的 cwd
                        _hulat → ${HULAT_DIR}        (symlink, 共享)
                        hulat-spec-issue<iid>/       ← Claude 写产物到这里
                        ...repo files...
                    log/
                        prompt.txt
                        claude_result.txt
                        acpx_raw.log
                        git_status.txt
                        git_diff.patch
                    attempt_state.json               ← per-attempt 元数据
                    summary.md                       ← 镜像到 GitLab issue 评论
                attempt-002/
                    ...
```

**关键设计点：**

- **agent 产物全部在仓库外** (`/data/openclaw_work/...`)，git 物理上看不到，杜绝意外 commit
- **每次 attempt 独立目录**：重试不覆盖前次产物，全部留盘供 audit
- **hulat_dir 单源共享**：每个 attempt 在 worktree 里做一个 symlink 指过去，零拷贝、改 hulat 一次全部生效
- **dispatcher 和 executor 状态分层**：campaign 级 / issue 级 / attempt 级三层 JSON

## 4. 分支与 MR 策略

采用**两分支模型**：

| 分支 | 角色 | trigger 字段 |
|------|------|------|
| `dev` | 干净基线。worktree fresh 模式从 `origin/dev` 切出，Claude 看不到任何过去 issue 的产物 | `dev_branch=dev` |
| `master` | 整合分支。所有 issue 的 spec 通过 MR 累积到这里 | `branch=master` |
| `issue/<iid>-auto-fix` | 单 issue 单远端分支（策略 A）。每次 attempt 用一条独立本地分支 `issue/<iid>-auto-fix-att<NNN>` 强制推送到这里 | 自动生成 |

**MR 策略 A：单 issue 单 MR + force-push**

- 第一次 attempt → 建 MR（source=`issue/<iid>-auto-fix`，target=`master`）
- 后续 attempt → MR 已存在，`create_mr.sh` 检测后直接复用，commit 通过强制推送进 MR
- MR description 自动以 `Closes #<iid>` 起头，merge 时 GitLab 自动关闭 issue
- agent **从不自动 merge** MR

**每 issue 输出隔离**：build_prompt.sh 通过 `# Rules` 硬约束告诉 Claude，所有产物只能写到 `hulat-spec-issue<iid>/` 子目录。多条 MR 在 master 上的 diff 永远是"新增独立目录"，不存在文件冲突。

## 5. Label 生命周期

```
                     ┌────────────────────────────────────┐
                     │                                    │
                     ▼                                    │
   (start) ──► doing ──► pr ──► done                      │
                │                                         │
                ▼                                         │
              blocked ──► doing  (retry after cooldown) ──┘
                │
                ▼
              failed   (retry exhausted, terminal)


   done ──► continue  (HUMAN review action; agent never sets continue itself)
              │
              ▼
            doing      (executor in continue mode, on next tick)
              │
              ▼
            pr ──► done   (or blocked / failed as usual)
```

七个标签 (`todo` / `doing` / `pr` / `done` / `blocked` / `failed` / `continue`) 在 agent 第一次跑时由 `ensure_labels.sh` 自动创建。

## 6. 续跑机制 (continue 模式)

reviewer 在 GitLab 上发现某条 issue 没真跑完时：

1. **写 comment 描述补救步骤** —— freeform，可以包含具体命令（如 `uv run robot --variable HEADLESS:True login.robot`），不用关心 agent 怎么解析
2. **把 label 从 `done` 改成 `continue`**

下一轮 dispatcher tick 自动：

1. `reconcile.sh` 探测到 `continue` label → 重新入队，per-issue state 设 `mode=continue`
2. spawn executor 时 trigger 带 `issue_mode=continue`
3. `prepare_attempt.sh` 在新的 `attempt-NNN/` 目录里建 worktree，BASE_REF = `origin/issue/<iid>-auto-fix`（保留前次 commit）
4. `build_prompt.sh` 把 issue notes 切两桶塞进 prompt：
   - **Past attempt summaries** —— 含 marker `<!-- uiautotester:attempt-summary -->` 的 agent 自发评论（前次跑过什么、改了什么、commit 是哪个、stderr 摘要）
   - **Reviewer comments** —— 其他所有非 system note，即人工补救指引
5. Claude 看完两段历史 + 现状 + 指引，继续干
6. 跑完后 `summarize_attempt.sh` 再 post 一条新 summary 到 issue，下次 continue 还能用

**重要边界**：reviewer 写的命令是给 **Claude** 执行的（通过它的 Bash tool），**executor 不会自己 bash 这些命令**。这条边界把"agent 编排"和"Claude 干活"分得很干净。

## 7. 配置文件

### `config/gitlab.env`

部署期 pin 一次，运行期只读：

```env
GITLAB_HOST=gitlab-b.pxsemic.tech:30000
GITLAB_API_PROTOCOL=http
```

trigger 里的 `gitlab_address` 是**校验值**，不一致直接 abort，从不静默换 host。

### Trigger 命令

```text
RUN_SCHEDULED_ISSUE_CAMPAIGN
group=<group>
project=<project>
branch=master
dev_branch=dev
hulat_dir=<hulat_dir>
gitlab_token=<token>
issue_min_iid=<min_iid>
issue_max_iid=<max_iid>
hourly_issue_quota=<quota>
max_runtime_minutes=<minutes>
blocked_retry_limit=<limit>
blocked_cooldown_ticks=<cooldown>
non_interactive=true
session_mode=per_issue
scheduling_mode=quota_carryover
blocked_policy=skip_and_retry
```

调度器每个 tick 都发同一条命令；agent 自己读 disk 状态决定该跑什么。

## 8. 多重防护规则（HARD RULE）

SKILL.md 顶部钉了五条**与生俱来不可违抗**的策略，目的是压住 LLM 的"自由发挥"倾向：

| 策略 | 防什么 |
|------|--------|
| **Concurrency Policy** | 防 dispatcher 看到 `hourly_issue_quota=3` 就同时 spawn 3 个 subagent；强制 1 个时刻 1 条 issue |
| **GitLab Access Policy** | 强制走 `glab` CLI，禁止 curl / wget / Python HTTP / 任何 SDK |
| **Source-of-Truth Policy** | 每 tick 必查 GitLab 真实 label，disk 缓存只是 cache，冲突时 GitLab 赢；reconcile 证据文件不写就当 tick 失败 |
| **No-Fallback Policy** | 失败就失败、按 blocked / failed 标记，禁止自创替代方案（如 acpx 失败改用 `claude` 直跑） |
| **Repo Cleanliness Policy** | leak guard 检查 staging 和 remote tree 不含 `openclaw_log/` `openclaw_state/` `_hulat`；中招直接 blocked |

每条都用 SKILL_VERSION 戳验证部署是否生效。

## 11. 当前已知边界

- **dispatcher 调度循环不会自停**：所有 issue 完成后，agent 每 tick 仍会唤醒做一次 reconciliation 后早退（成本几秒）。要彻底停只能去 OpenClaw 关 scheduled task。
- **OpenClaw 平台层并发限制需运维确认**：当前 `Subagent Concurrency Policy` 是 prompt 级硬规则，已加 spawn-slot 兜底设计，但 OpenClaw 是否提供原生 max-parallel-subagents=1 配置没确认；建议运维侧落实。
- **Claude Code 路径输出冲突**：如果 issue 描述里写绝对路径（如 `/data/openclaw/.../ifp-result/`），会和 `# Rules` 段的 `hulat-spec-issue<iid>/` 约束冲突，需要项目方在 issue 文案里改用相对路径或省略路径。
