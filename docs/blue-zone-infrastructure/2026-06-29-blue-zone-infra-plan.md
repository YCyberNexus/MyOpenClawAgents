# 蓝区基础设施梳理（盘点 · 选型 · 优先级）

> 版本：v0.1（草案，待评审）　日期：2026-06-29　牵头：研发平台
> 配套图：`openclaw_architecture.drawio`（OpenClaw 多区协同研发架构）

---

## 0. 一页概要（给领导先看这页）

**背景。** 我司采用绿 / 黄 / 蓝**三层网络隔离**。蓝区是「智能研发与交付区」，跑着一整套 OpenClaw AI Agent（需求接入 → 建模 → 建 Issue → 编码 → 测试 → PR → Git），是隔离级别最高、最贴近核心代码与生产的一层，**基本与公网断开**。本梳理回答一个问题：要让这套 AI 自动研发 + 交付流水线在蓝区里**自给自足地跑起来并可治理**，还需要哪些基础服务。

**核心结论（5 条）。**
1. **蓝区断网，"私服"不是可选项而是前提。** 凡是平时从公网拉的——容器镜像、各语言依赖包、操作系统包、甚至 LLM 模型权重——蓝区里都要有内网等价物。这次盘出 **12 个域、约 78 项**基础服务，现状一律按"待建"处理。
2. **最该先钉死的是"制品摆渡管线"（A 域），它是所有私服的源头。** 大多数盘点直接列各种私服，却漏了"私服里的东西从哪来"。必须先有一条**受控、单向、可审计**的外部→蓝区摆渡通道，否则镜像库 / 包私服 / 模型仓库都是空壳。
3. **自托管 LLM 推理是 Agent 平台的命脉（F 域，P0）。** 蓝区没有公网 API，整套 Agent 离开内网大模型就停摆。这是和"传统研发隔离区"最大的不同点，需要 GPU 资源与模型摆渡配套。
4. **在隔离区里引入 LLM 本身是一把双刃剑，要专门管控数据出域（I 域）。** LLM 网关的出入参、摆渡通道的出站方向，都是核心代码 / 数据潜在的外泄口。隔离做得再严，也要在这两处加 DLP 与审计，否则等于开了后门。
5. **有一批"零号基础设施"必须最先建**：内网 DNS、NTP 时间同步、内网 CA / 证书、对象存储、统一身份。它们被几乎所有上层服务依赖，排期上要排在最前面。

**优先级总览。**

| 优先级 | 含义 | 覆盖的域 / 代表服务 |
|---|---|---|
| **P0 底座** | 没有它 Agent 或流水线根本起不来 | A 制品摆渡、F1/F2 内网 LLM 推理 + 网关、E2 对象存储、E3 数据库、C1 容器镜像库、G1 密钥、K1/K2 DNS+NTP、G3 内网 CA |
| **P1 打通** | 让多语言项目能编译 / 测试 / 打包 / 交付 | C 各语言私服 + OS 包镜像、D1/D4 CI + 测试环境池、E1 容器编排、B Git、H4/H5 扫描门禁、G2 身份 |
| **P2 治理** | 可观测、合规、备份、签名、审计（隔离区其实也重要，可打通后补） | H 可观测 / 审计、I 数据安全 / DLP、J 备份容灾、G4/G5 签名 + 准入、L 运维接入 |

**建设节奏建议**：第一期落 P0（零号基础设施 + 摆渡 + LLM 底座 + 镜像库）；第二期落 P1（多语言私服 + CI + 测试环境，打通端到端）；第三期落 P2（观测 / 安全 / 备份 / 合规收口）。

## 1. 范围与口径

- **覆盖层面**：端到端——① 研发交付流水线依赖（镜像 / 包 / 制品 / Git / CI），② AI Agent 平台运行底座（LLM / 模型 / 向量 / 调度），③ 横切治理（安全 / 数据 / 备份 / 网络 / 运维）。
- **技术栈**：Lua(LuaRocks)、Java(Maven/Gradle)、Python(pip)、Node(npm)、Go、C/C++。决定了 C 域要建哪些语言私服。
- **现状口径**：本轮**默认全部待建**，逐项给"是什么 / 为什么 / 选型 / 优先级 / 落地要点"。实际已有的服务，评审时标注"已有"即可剔除或降级。
- **选型原则**：① 开源优先、可离线部署；② 能一套覆盖多需求的优先（减少运维面）；③ 来源可信、可签名验签；④ 如有信创 / 等保要求，可替换为对应国产组件（见 §16 待确认）。

### 1.1 主链路 → 基础服务映射（导览）

| 流水线阶段 | 对应 Agent | 强依赖的基础服务 |
|---|---|---|
| 需求接入 | req_dispatcher | Channel 跨区通道(A3)、身份(G2)、内网 LLM(F1/F2)、反代/DNS(K1/K3) |
| 建模 / 建 Issue | emcp / git_issuer | Git(B)、LLM(F)、状态存储 DB+KV(E3/E4)、密钥(G1) |
| 编码 | Coding Agent | LLM(F)、Git(B)、各语言私服(C)、密钥(G1) |
| 测试 | req_executor / acpx | 测试环境池(D4)、容器镜像库(C1)、OS 包 + 语言私服(C)、对象存储(E2) |
| PR 审核门禁 | — | SAST/secret 扫描(H5)、漏洞扫描(H4)、制品签名(D5/G4)、CI(D1) |
| 蓝 → 黄 Git 同步 | — | 单向同步通道(A2) |
| 全程横切 | 所有 | 可观测(H)、密钥/身份(G)、数据安全/DLP(I)、备份(J)、网络/DNS/NTP/CA(K) |

## 2. 梳理框架

按基础设施**域**分层盘点（不重不漏、利于落地分工），每项固定五要素：**是什么 / 为什么蓝区需要 / 推荐选型 / 优先级 / 落地与摆渡要点**。文档开头已给主链路映射表作为另一视角导览。

优先级口径：
- **P0 底座**：缺它 Agent 或流水线根本跑不起来。
- **P1 打通**：让多语言项目能编译 / 测试 / 打包 / 交付。
- **P2 治理**：可观测、扫描、审计、合规、备份、签名。

12 个域：A 跨区边界与制品摆渡 · B 源码与协作 · C 制品与依赖私服 · D 构建与 CI/CD · E 运行底座 · F AI Agent 平台底座 · G 安全·密钥·身份 · H 可观测·扫描·审计 · I 数据安全与合规 · J 备份·容灾·高可用 · K 基础网络与平台服务 · L 运维接入与变更管理。

---

## 3. A 域 · 跨区边界与制品摆渡（air-gap）

> **隔离区的灵魂。** 蓝区一切私服的"货源"都来自这里；出站方向则是数据外泄的第一道闸。建议作为第一优先级钉死流程与设备。

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| A1 | **入站制品摆渡管线** | 把外部的容器镜像 / 各语言依赖包 / OS 包 / 模型权重 / 工具二进制，经受控通道单向同步进蓝区；含申请—审批—安检—入库—台账闭环 | 网闸 / 数据二极管 + skopeo 同步镜像、各私服"远程仓库离线导入"、单向 rsync；统一摆渡门户记录台账 | **P0** |
| A2 | **蓝 → 黄 Git 单向同步** | 架构图已画的代码镜像，底层需单向复制机制，黄区只读不回写 | git mirror push over 单向网关 / 定时单向同步作业 | P1 |
| A3 | **跨区消息通道 Channel 承载** | 绿→蓝需求、蓝→黄知识查询的受控中继（逻辑已有，盘点其底层网关 / 代理） | 受控反向代理 + 协议白名单 | P1 |
| A4 | **出站策略与网闸/单向网关** | 控制蓝区能否出网、出什么；与 I 域 DLP 联动 | 单向网关设备 + 出站白名单 + 内容过滤 | **P0** |
| A5 | **摆渡内容安全基线** | 进来的镜像 / 包必须过病毒 + 恶意包 + 供应链投毒检测才入库（与 H4 联动） | Trivy/ClamAV + 来源校验 + 哈希/签名核验 | P1 |

💡 **落地要点**：A1 决定建设节奏——它没通，C / F 域所有私服都没法填充内容。建议先定"摆渡审批流 + 安检基线 + 台账"，工具其次。**强约束**：A1 入站与 A4 出站必须物理 / 逻辑分离，出站默认拒绝。

|---|---|---|---|---|

## 5. C 域 · 制品与依赖私服（你的重点）

> 多语言私服。建议**两件套打底**：Harbor（容器镜像）+ Nexus/Artifactory（多语言通用制品库），覆盖绝大多数语言；少数特殊的（Lua / C++）单独补。

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| C1 | **容器镜像仓库（"本地 DockerHub"）** | 你明确要的；基础镜像 + 自建镜像，带扫描 / 签名 / 复制 / 配额 | **Harbor**（首选，自带 Trivy 扫描 + Cosign 签名 + 复制）/ Nexus docker repo | **P0** |
| C2 | **统一制品库（多格式）** | 一套托管 Maven/PyPI/npm/Go/raw 等，省运维 | **Nexus Repository OSS** / JFrog Artifactory(商业) | **P0** |
| C3 | **LuaRocks 私服** | Lua 模块（OpenResty/Nginx/Redis/Skynet 场景） | luarocks 自建 server / 上游 mirror，或用 Nexus raw 托管 `.rock` | P1 |
| C4 | Maven / Gradle 私服 | Java jar 依赖 | Nexus maven(hosted+proxy) | P1 |
| C5 | PyPI 私服 | Python 包 | devpi / Nexus pypi | P1 |
| C6 | npm 私服 | Node 包 | Verdaccio / Nexus npm | P1 |
| C7 | Go module proxy | Go 依赖 | Athens / JFrog go / Nexus go | P1 |
| C8 | C/C++ 依赖 | 系统组件 / Nginx 模块 / 扩展库 | Conan 私服 + vcpkg binary cache + 源码 tarball 镜像 | P1 |
| C9 | **OS 包镜像** | apt/yum/apk + 基础镜像镜像，C/C++ 编译与所有容器构建都依赖 | apt-mirror/aptly、reposync、本地 base image 镜像 | **P1** |
| C10 | 通用 raw/generic 仓库 | 工具二进制、SDK、IDE 插件、字体等杂项 | Nexus raw | P1 |
| C11 | 制品保留 / 清理 / 配额策略 | 控制存储膨胀、保留可追溯版本 | 各库 retention policy | P2 |

💡 **落地要点**：所有 C 域私服的内容都来自 A1 摆渡（proxy 仓库在内网指向"摆渡缓存"，而非公网）。LuaRocks(C3) 与 C/C++(C8) 是最容易被通用制品库漏掉、需单独验证的两块。

## 6. D 域 · 构建与 CI/CD

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| D1 | CI/CD 引擎 | 编译 / 测试 / 打包 / 发布流水线 | GitLab CI（与 B 一体）/ Jenkins | P1 |
| D2 | 构建 Runner 池 | 多语言构建执行器（含各语言构建镜像） | GitLab Runner(K8s executor) | P1 |
| D3 | 构建缓存 | 依赖缓存 + Docker layer + ccache/sccache(C++) + Gradle/Maven 缓存 | 对象存储(E2) 后端 + 各缓存组件 | P2 |
| D4 | **测试环境编排 / 沙箱池** | 动态拉起被测服务 + 依赖(DB/Redis/MQ)，承载 emcp 的 env_slot | K8s namespace / docker-compose 池 + 环境模板 | **P1** |
| D5 | 制品签名 + SBOM | 可信交付、可追溯成分 | cosign(签名) + syft(SBOM) | P2 |
| D6 | 流水线模板库 | 流水线即代码、跨项目复用 | GitLab CI templates | P2 |
| D7 | 质量门禁集成 | 测试覆盖 / 扫描结果阻断 PR | 与 H4/H5 集成到门禁 | P1 |

## 7. E 域 · 运行底座

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| E1 | 容器编排 / 计算平台 | 承载 agents、私服、测试环境的统一算力底座 | Kubernetes（首选）/ 裸 Docker（小规模） | P1 |
| E2 | **对象存储（S3 兼容）** | 制品 / 模型权重 / 日志 / 备份 / 缓存的统一底座，被众多服务依赖 | **MinIO** | **P0** |
| E3 | **关系型数据库** | GitLab/Harbor/Nexus/Agent 状态 的元数据 | **PostgreSQL**（HA） | **P0** |
| E4 | 缓存 / KV / 队列 | Agent 状态、锁、异步任务（对应 flock/state/cron） | Redis + （如需）RabbitMQ/NATS | P1 |
| E5 | 配置中心 / 服务发现 | 多服务配置与发现 | K8s ConfigMap+Service / Consul/Nacos | P2 |
| E6 | 共享 / 块存储 | PV、模型权重大容量盘 | Ceph / NFS / CSI | P1 |

💡 E2 对象存储是"零号基础设施"之一——制品库、模型仓库、备份、日志后端都建议挂在它上面，先建。

## 8. F 域 · AI Agent 平台底座（端到端的另一半）

> 蓝区断网，没有任何公网 LLM API。这一域是整套 OpenClaw Agent 的命脉，且需要 GPU 与模型摆渡配套。

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| F1 | **内网 LLM 推理服务** | 自托管开源大模型，给所有 Agent 供算力 | **vLLM / SGLang** 部署 Qwen / DeepSeek / GLM 等；GPU 资源池 | **P0** |
| F2 | **LLM 网关（OpenAI 兼容）** | 统一鉴权 / 路由 / 限流 / 配额 / 计费 / 审计（对应 hourly 配额） | **LiteLLM / one-api** | **P0** |
| F3 | **模型权重仓库 / 分发** | 权重也要经 A1 摆渡进来，需托管 + 校验 + 版本 | MinIO(E2) + 校验脚本 / 内网 HF mirror | **P0** |
| F4 | 向量数据库 | RAG 与 Agent 长期记忆 | Milvus / Qdrant / pgvector | P1 |
| F5 | Embedding / Rerank 服务 | 检索召回与重排的本地推理 | 自托管 bge/embedding 模型 | P1 |
| F6 | Agent 运行时与调度 | OpenClaw agents 承载 + 状态存储 + cron（逻辑已有，盘其依赖） | 容器化部署 + E3/E4 + 定时器 | P1 |
| F7 | Prompt/Skill/Agent 配置版本库 | SOUL/AGENTS/USER/SKILL 的版本化与分发 | Git(B) 管理 + 发布流程 | P1 |
| F8 | GPU 资源调度 / 共享 | 多 Agent / 多模型共享有限 GPU | K8s device plugin / MIG / 时分复用 | P1 |
| F9 | LLM 可观测与评测 | 调用 trace、回放、质量评估（进阶） | Langfuse 内网版 / 自建 | P2 |

💡 **落地要点**：F1+F2+F3 是不可分割的 P0 三件套。模型选型与 GPU 预算需领导拍板（见 §16）。F2 网关是把"成本审计 + 数据出域管控(I4)"落到一处的关键卡点。

|---|---|---|---|---|

## 11. I 域 · 数据安全与合规

> 在隔离区里引入 LLM，最大的悖论是：你为了智能引入了一个可能成为"出口"的组件。本域专治"核心代码 / 数据不出域"。

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |
|---|---|---|---|---|
| I1 | 数据分类分级与标记 | 哪些是核心 / 敏感 / 可外发，定级才能管控 | 分级规范 + 标签 | P1 |
| I2 | 静态加密 | 制品库 / 对象存储 / DB 落盘加密 | 存储层 / KMS 加密(G1) | P2 |
| I3 | 传输加密 | 全内网 TLS（依赖 G3 CA） | mTLS / 服务网格 | P1 |
| I4 | **DLP 数据防泄漏** | 重点在 A4 出站口 + F2 LLM 网关出入参，防核心代码 / 数据经摆渡或 LLM 外泄 | 网关侧内容检测 + 出站 DLP 策略 | **P1** |
| I5 | 敏感信息检测 | 代码 / 日志里的密钥 / 个人信息（与 H5 联动） | gitleaks / 自定义规则 | P1 |
| I6 | 数据出域管控 | 蓝区数据外发的审批与拦截（与 A4 联动） | 出站白名单 + 审批流 | P1 |
| I7 | 合规基线映射 | 等保 / 行业要求映射 + 证据留存 | 合规清单 + H7 审计证据 | P2 |

💡 I4 + I6 是这套"AI 隔离研发"模式的安全底线，建议与 A4、F2 一起设计，作为同一道闸的两面。

## 13. K 域 · 基础网络与平台服务（零号基础设施）

> 这些服务被几乎所有上层组件依赖，"看不见但缺了全塌"。排期上最先建。

| 编号 | 服务 | 是什么 / 为什么 | 推荐选型 | 优先级 |

## 15. 建设路线（分期）

**第一期 · P0 底座（先让平台能呼吸）**
零号基础设施（K1 DNS、K2 NTP、G3 CA、E2 对象存储、E3 DB、G1 密钥）→ A 制品摆渡管线 → C1 容器镜像库 → F1/F2/F3 内网 LLM 推理 + 网关 + 模型仓库。
**里程碑**：Agent 能在内网调通大模型，镜像能摆渡入库。

**第二期 · P1 打通（端到端跑起来）**
C 各语言私服 + C9 OS 包镜像 → B Git → D1/D2/D4 CI + Runner + 测试环境池 → E1 容器编排 → F4/F6 向量库 + Agent 运行时 → G2 身份 → H4/H5 扫描门禁。
**里程碑**：需求 → 编码 → 测试 → PR → 同步 全链路在蓝区闭环跑通。

**第三期 · P2 治理（可观测 / 安全 / 合规收口）**
H 可观测 + H7 审计 → I 数据安全 + DLP → J 备份容灾 → G4/G5 签名 + 准入 → L 运维接入 + IaC。
**里程碑**：合规、可审计、可灾备重建。

## 16. 风险与待领导 / 你确认的关键决策

1. **隔离强度**：蓝区是"完全物理隔离 + 人工摆渡"，还是"允许受控有限出网"？直接决定 A 域是网闸方案还是代理方案。
2. **自托管模型选型与 GPU 预算**：用哪个开源模型档位（决定编码 / 测试质量）、配多少 GPU——F 域 P0 的前置。
3. **信创 / 等保要求等级**：是否要求国产化替换（CPU/OS/数据库/中间件）、等保几级——影响 C/E/G 全域选型与 I7 合规。
4. **制品库选型**：Nexus OSS（免费、够用）vs Artifactory（商业、功能全）——影响 C 域投入。
5. **摆渡审批与责任**：A1 摆渡的审批人、SLA、安检红线由谁定——这是流程问题，先于工具。
6. **现状校正**：本文档默认全部待建；评审时请逐项标注蓝区实际已有的服务，以收敛真实缺口与预算。
