# UiAutoTester Agent 工作内容文档

> 时间范围：项目启动 → `SKILL_VERSION=2026-04-25.2`
> 目的：按时间线说明对该 agent 做了哪些工作、每一步要解决什么问题、改了什么、效果如何

---

## 项目起点

最初 workspace `workspace-UiAutoTester/` 是一个粗糙的 OpenClaw agent 模板：

- 单一 `SKILL.md` 把所有规则、bash 代码、schema 全塞在一个文件里
- agent 只是 hello world 级的"读 issue → 调 Claude Code → 推分支 → 开 MR"
- 没有版本戳、没有真值校验、没有失败兜底、没有续跑机制
- agent 经常出现以下行为偏差：
  - 把 `hourly_issue_quota=3` 误读为"同时起 3 个 subagent"
  - 把 `hulat_dir` 误当成 working / 状态目录
  - issue 在 GitLab 上从 `done` 改回 `todo` 后 agent 仍说 "all completed"
  - acpx 调用失败后自创替代方案（改用 `acpx claude command` / 直跑 `claude`）
  - issue 分支里混进其他 issue 的产物（`git add -A` 扫到遗留文件）

下面按时间顺序记录每一步改造。

## 第 1 阶段：基础硬规则补全

### 1.1 强制 issue 串行执行

**问题**：agent 把 `hourly_issue_quota=3` 当成并发度，一次起 3 个 subagent 处理 3 条 issue。

**改动**：SKILL.md 顶部加 **Concurrency Policy (READ FIRST — HARD RULE)**：

- 任意时刻最多 1 个 active subagent
- `hourly_issue_quota=3` 是"顺序处理 3 个"，不是"并发 3 个"
- `sessions_spawn` 必须独占 tool-call batch
- 必须阻塞等回复才能 spawn 下一个

**Commit**: `7c5a6ec issue强制顺序执行` / `7a94b0c 重组并加固 Subagent 并发限制条款`

---

### 1.2 GitLab 真值校验 (reconciliation)

**问题**：reviewer 把 GitLab 上某条 issue 从 `done` 手动改回 `todo`，agent 下一轮仍按 disk 缓存的 `completed` 早退，issue 不会重跑。

**改动**：

- 新增 dispatcher 算法 Step 3 **Trigger-input override**：trigger 里所有标量（IID 范围、quota 等）每次必须覆盖 disk 上的旧值
- 新增 Step 4 **GitLab-truth reconciliation MANDATORY**：每次唤醒必须调 GitLab API 查每个 IID 的真实 label，写到 `/data/.../openclaw_log/dispatcher/reconcile-<ts>.json` 当证据文件
- 新增 Step 5 **Early-return gate**：早退必须同时满足"证据文件已写 + 没 IID 重开 + unfinished 空 + status=completed"

**效果**：reviewer 改 label 后，下一轮自动从 `completed_iids` 移除该 IID、塞回 `unfinished_iids`、备份并重置 per-issue 状态、`campaign_status` 强制回到 `running`。

**Commit**: `eef37a3 新增 GitLab-truth reconciliation` / `78184c5 input-range override + reconciliation 强化`

---

### 1.3 hulat_dir 误用修复

**问题**：模型把 trigger 里的 `hulat_dir=/data/openclaw/bu_data/px_hulat/` 当成了状态根目录，把 `openclaw_log/` 和 `openclaw_state/` 写到了 hulat 目录下。

**改动**：SKILL.md `## Paths` 段把"hulat_dir 不是工作目录、不是状态根、不是 repo 路径，仅是 prompt 字符串"写成显式硬规则。

**Commit**: `03f11d0 修复模型把 hulat_dir 当成了状态根目录`

---

### 1.4 No-Fallback Policy

**问题**：acpx 调用失败、Claude Code 中途出错、glab 报错、git push 被拒时，agent 自创替代方案（改 acpx 模式 / 跑 `claude` / 用 curl / `git push --force` / 手工补 patch），导致行为不可控。

**改动**：两个 SKILL.md 顶部加 **No-Fallback Policy (READ FIRST — HARD RULE)**：

- 脚本退出非零 → 不得手抄实现
- glab 不能做 → 不得回退 curl
- acpx 失败 → 不得改用 `acpx claude command` / `-s` / `--no-wait` / 直跑 `claude` / 换 LLM / 手工改 repo
- `git push` 被拒 → 不得 `--force` / `rebase` / 换分支名
- guard 报泄漏 → 不得手工 `git rm` 后重 stage

宣言：**"a clean controlled failure is strictly better than an unsupervised alternative attempt."**

**Commit**: `059b1a2 加 No-Fallback Policy`

---

### 1.5 部署期 GitLab host pin

**问题**：每次 tick 都从 trigger 字符串 `${GITLAB_ADDRESS}` 用 `sed` 解析 host，长时间运行可能因变量丢失 / 转义出错而静默换地址。

**改动**：

- 新建 `workspace-UiAutoTester/config/gitlab.env`，pin 一次：`GITLAB_HOST` + `GITLAB_API_PROTOCOL`
- 两个 `glab_auth.sh` 改为 source pin 文件、用 trigger 的 token 刷新而不解析 host
- trigger 里的 `gitlab_address` 改为可选校验值，对不上 exit 13、不静默换 host

**Commit**: `733fdd6 GitLab host 固定到部署级 config/gitlab.env` / `9e45120 gitlab.env 填实际值，trigger 里 gitlab_address 改为真正可选`

---

### 1.6 强制走 glab CLI

**问题**：agent 默认用 curl 调 GitLab REST API，命令拼装易出错。

**改动**：

- SKILL.md 加 **GitLab Access Policy (READ FIRST — HARD RULE)**：仅允许 `glab` CLI；禁止 curl / wget / Python HTTP / 任何 HTTP 库 / 任何不在 references 中的 glab 子命令
- 把所有 GitLab 操作命令做成查表式：`E1` 读 issue / `E2` 列 labels / `E4` add label / `E5` remove label / `E6` list MR / `E7` create MR / `E9` post issue note 等
- 在 `references/glab_commands.md` 中给出**逐字 verbatim** 的命令模板，模型直接抄

**Commit**: 在多个分阶段提交里持续加固

## 第 2 阶段：可维护性重构

### 2.1 SKILL.md 拆分（scripts/ + references/）

**问题**：SKILL.md 一度涨到 450+ 行，包含大量 bash 代码块和 schema，维护性差，agent 进 SKILL 后 context 也大。

**改动**：把每个 SKILL 拆成三层：

```
skills/<skill>/
├── SKILL.md       (政策 + 算法骨架，~150 行)
├── scripts/       (可执行脚本，每个职责单一)
└── references/    (固定知识库 —— path schema / state schema / 命令清单 / lifecycle)
```

效果：SKILL.md 减少 50% 以上，scripts 各司其职、可独立测试，references 在需要时按需 Read。

**Commit**: `7f35b11 拆分 SKILL.md：scripts/ 放可执行脚本，references/ 作为知识库`

---

### 2.2 SKILL_VERSION 版本戳

**问题**：覆盖部署后无法确认服务器上加载的是不是新版 SKILL。

**改动**：

- 两个 SKILL.md 在 front-matter `description` 头部加 `[SKILL_VERSION=YYYY-MM-DD.N]`
- 在文档正文顶部 bold 一行
- 强制 dispatcher / executor 在每次回复 JSON 里都带 `"skill_version": "..."`
- executor 还要把版本号写进 `${ISSUE_STATE_FILE}.skill_version`

**Commit**: `6f81124 skill description 加上版本号`（之后每次结构变更都跟版本递增）

### 2.3 SOUL.md / AGENTS.md 加固并发限制

**问题**：仅靠 SKILL.md 写"串行" prompt 级别压不住，需要在多个层次重复声明。

**改动**：

- SOUL.md 新增 `## Subagent Concurrency Policy (READ FIRST — HARD RULE)`，置于 `## Roles` 之后最显眼位置
- AGENTS.md `## Runtime Limit` 重写，加 `### Behavioral commitments`（与 SOUL.md 镜像）+ `### Enforcement` 三层落实方案：(1) OpenClaw 平台层旋钮（首选）/ (2) workspace spawn-slot 兜底 / (3) prompt 规则（兜底）

**Commit**: `7a94b0c 重组并加固 Subagent 并发限制条款`

## 第 3 阶段：业务流程加强

### 3.1 MR merge 后自动关 issue

**问题**：agent 标 done 后 issue 仍保持 opened，需要人工去 GitLab close。

**改动**：

- `scripts/create_mr.sh` 让 auto-generated MR description 第一行加 `Closes #<iid>`
- GitLab 看到关闭关键字会在 MR 真正 merge 时自动 close 关联 issue
- SKILL.md 同步加规则：executor **不得**自行 close issue（不得用 `state_event=close` / `glab issue close`）。close 是 GitLab 干的事

**Commit**: `5c28564 MR description 加 "Closes #<iid>"`

---

### 3.2 续跑机制 (continue 标签)

**业务背景**：项目方反馈"agent 收到 acpx 返回就标 done，但 Claude Code 实际可能因为环境错误等没真跑完"。需要人工兜底。

**改动**（横跨两次 commit）：

**第一步**（commit `b275917`）—— 引入 `continue` label：

- `scripts/ensure_labels.sh` 加 `continue` 进必备 label
- `reconcile.sh` 输出 evidence 加 `needs_continue` 字段
- dispatcher Source-of-Truth Policy 处理 `needs_continue`：从 `completed_iids` 移除 → 加进 `unfinished_iids` → 备份旧 state.json → 写新 state，`mode=continue`
- spawn 时 trigger 带 `issue_mode=continue`
- executor 改写 `prepare_branch.sh`：continue 模式下 BASE_REF = `origin/issue/<iid>-auto-fix`，不再从 master 重切；远端分支不存在自动降级 fresh，并写 `mode_downgraded_from="continue"` 留 audit
- `references/label_lifecycle.md` 加 `done → continue → doing` 转移、`continue → doing` 操作

**第二步**（commit `9a7bb26`）—— 让 agent 真正能"读懂"补救步骤：

- 之前 executor 只读 issue 的 title/description/labels，不读 comments，reviewer 写的补救步骤等于石沉大海
- 新增 `scripts/build_prompt.sh`：在 continue 模式自动调 glab E1b 抓 issue notes，把内容**原文**塞进 prompt 的 `# Reviewer comments` 段
- 新增 `references/continue_mode.md` 写明 reviewer 契约：先写 comment 描述补救步骤、再翻 label
- 显式声明 **executor 不得自己 bash 评论里的命令** —— 评论是给 Claude 看的，由 Claude 通过 acpx 进程的 Bash tool 去执行。这条边界把"agent 编排"和"Claude 干活"分得很干净

**领导原话回应**：把具体命令（如 `uv run robot --variable HEADLESS:True login.robot`）写在 issue 评论里、agent 读 issue 重跑——**不要**把命令写死在 SKILL.md。理由：每条 issue 失败原因不同，硬编码进 SKILL 既僵化又跟通用化原则冲突。当前实现完全贯彻这个意见。

## 第 4 阶段：文件隔离 + worktree 改造（最大一次重构）

### 4.1 起因

随着 issue 处理量增多，三个问题暴露：

1. **不同 issue 的产物互相串扰**：issue #6 的 MR 分支里出现了 #2/#5/#7 等其他 issue 的 log 文件夹
   - 根因 1：log/state 路径在 repo 工作树里（`${REPO_PATH}/openclaw_log/`），git 天然能扫到
   - 根因 2：executor 用 `git add -A`，把所有遗留 log 文件夹一起 staged
   - 根因 3：切分支前没清理工作树
2. **同一 issue 多次重试无法物理隔离**：每次重试覆盖前次产物，无法 audit "上次跑成什么样"
3. **每条 issue 一个分支扩展性差**：分支数随 issue 数线性增长，git 维护负担

### 4.2 重构方案（commit `9ab97d2`）

完全重写磁盘布局：

```
/data/${PROJECT}/                            ← 主仓库（不再编辑工作树）
/data/openclaw_work/${PROJECT}/              ← agent 全部产物，仓库外
    openclaw_state/                          campaign 级
    openclaw_log/dispatcher/                 dispatcher 级
    issues/issue-<iid>/
        state.json                           cross-attempt
        attempts/attempt-<NNN>/
            worktree/                        git worktree (Claude 的 cwd)
                _hulat → ${HULAT_DIR}        symlink (零拷贝 + 单源)
            log/                             prompt/result/raw/git_status/git_diff
            attempt_state.json
            summary.md
```

**关键设计：**

| 设计点 | 做法 | 防什么 |
|--------|------|--------|
| **agent 产物全在仓库外** | log/state 路径搬到 `/data/openclaw_work/...` | git 物理上看不到，杜绝意外 commit |
| **每次重试新建 attempt-NNN/** | 编号单调递增，前次产物全保留 | 可 audit、可对比、不被覆盖 |
| **每个 attempt 一个 git worktree** | `git worktree add -b <local> <dir> <base>` | 多 attempt 物理隔离，主仓库不被污染 |
| **hulat_dir 单源 + symlink** | `ln -sfn ${HULAT_DIR} _hulat` 进 worktree | 零拷贝、改 hulat 一次全 attempt 生效 |
| **`_hulat` 进 `.git/info/exclude`** | 写 worktree-local exclude，不动仓库 .gitignore | symlink 不会被 commit；仓库历史不变 |
| **leak guard 三重验证** | stage 前扫 / commit 前 grep / push 后 ls-tree | `openclaw_log/` `openclaw_state/` `_hulat` 一律拦死 |

策略 A：单 issue 单远端分支 `issue/<iid>-auto-fix`，每个 attempt 用独立本地分支 `issue/<iid>-auto-fix-att<NNN>`，commit 后 `--force-with-lease` 推到远端固定分支。MR 永远只有一条，自动复用。

新增脚本：

- `prepare_attempt.sh`（替换旧 `prepare_branch.sh`）—— 切 worktree + symlink + exclude
- `summarize_attempt.sh` —— 每个 attempt 完成（含失败）都生成 `summary.md` 并 POST 到 GitLab issue 当评论；comment 用 marker `<!-- uiautotester:attempt-summary v1 attempt=NNN -->` 包起来便于下次区分

`build_prompt.sh` 在 continue 模式下把 issue notes **切成两桶**：

- 含 marker 的 → "past attempt summaries"
- 不含 marker 的 → "reviewer comments"

两桶都灌进 prompt，让 Claude 看到完整历史 + 人工指引。

### 4.3 两分支模型 + 输出目录隔离（commit `d699c18`）

**业务背景**：核对了项目方提供的"基于 worktree 的 openclaw 执行方案"文档后，发现两个对齐缺口。

**A. 两分支模型**

- 单分支 (`master`) 模式下 worktree 从 master 切出，里面已有过去所有 issue 的 spec，Claude 启动时上下文就被污染
- 引入 `dev_branch` 字段：`branch=master` 是整合 / 目标分支（MR 开向这里），`dev_branch=dev` 是干净基线（fresh 模式 worktree 从这里切）
- `prepare_attempt.sh` fresh 模式 BASE_REF 改为 `origin/${DEV_BRANCH}`，continue 模式仍是 WORK_BRANCH
- 加 BASE_REF 存在性 sanity check（exit 5），避免空 ref 建 worktree
- `build_prompt.sh` fresh-mode 前言告知 Claude "你在干净基线上，master 上有过去 issue 的累积但你看不到"

**B. 每 issue 输出目录隔离**

- `build_prompt.sh` `# Rules` 段加硬约束：所有 spec / report / artifact 必须写到 `hulat-spec-issue<IID>/` 子目录
- 多条 MR 在 master 上的 diff 永远是"新增独立目录"，不存在文件冲突，merge 始终干净

## 待项目方 / 运维侧确认的事项

1. **OpenClaw 平台层并发限制**：建议运维确认是否有原生 `max_parallel_subagents=1` 的配置项，落实后可拆 spawn-slot 兜底

3. **Issue 文案规范**：项目方在 issue 描述里**不要写绝对输出路径**（如 `/data/openclaw/.../ifp-result/`），让 agent 的 `# Rules` 段（写到 `hulat-spec-issue<iid>/`）一锤定音
4. **下游消费者**：原本读 `/data/openclaw/.../ifp-result/` 的工具，改读 master 上的 `hulat-spec-issue<iid>/<A>/` 目录
