# 主状态机汇报讲稿（口语版）

> 对应图：`statemachine.v2.drawio` 第 1 页「Issue 主状态机」。
> 全程约 6–8 分钟。〔方括号〕里是给自己看的提示，念的时候跳过。

---

## 〔开场 · 约 40 秒〕

各位领导、各位同事,我来汇报一下这套自动化测试 Agent 的状态机。上次评审大家给了挺多意见,这一版我基本都照着改了。

先说最大的一个变化:**我把原来挤在一张图里的东西,拆成了两台状态机。** 一台是「Issue 主状态机」,就是大家现在看到的这张,它只画**人需要看到、需要动手**的状态;另一台是「调度器和执行状态机」,画的是机器自己跑的那些内部步骤,放在第二页了。

今天咱们重点看第一页就行——因为这一页才是跟人打交道的那一面,第二页是给开发和运维看的。

## 〔一句话原则 · 约 30 秒〕

这张图其实就守一条原则:**人看到一个状态,马上就知道该怎么处理。**

所以图上每个框,要么是「你得盯着的」,要么是「你得动手的」。那些不用人管的自动步骤——准备代码、起子任务、提交、推送、建 Wiki、开 MR——我全从这张图里拿掉了,沉到第二页去了。这样主图就特别干净,一眼能看完,不会被一堆技术细节糊住。

## 〔正常路径 · 约 1 分 30 秒〕

咱们从左边看起。

**第一个改动:这台状态机的起点,是一个标签。** 〔指最左边的圆点 → QUEUED〕以前我还画了用户在网页上建 issue、填表单那一套,其实那跟自动化没关系。现在起点很简单——**人在 issue 上打个入口标签**,比如 `todo`、`new`,issue 就进了 `QUEUED`,也就是「排上队了,等着处理」。这一步是人干的,也是整台机器的开关。

打完标签它就排队。调度器隔一阵扫一遍,挑中它,就进 `AUTO_RUNNING`,对应 GitLab 上的 `doing`。〔指 AUTO_RUNNING〕这个框我故意写得特别简单,意思就一句:「agent 正在干活,你先别管,等结果就行。」**至于它里头到底怎么干的,那是另一台状态机的事,在第二页**,这儿我不展开——就是不想让两台状态机又混到一块儿去。

等它干完、代码提交了、MR 也开好了,issue 就到 `AWAITING_REVIEW`,「等人评审」。〔指 AWAITING_REVIEW〕**这儿有个这次的新改法:这个状态现在只挂一个 `pr` 标签。** 以前是 `done` 和 `pr` 俩一起挂着,这次改成 `pr` 把 `done` 顶掉——MR 一建好打上 `pr`,`done` 就摘了。这么改的好处是,一个 issue 同一时刻就一个干干净净的状态标签,不会俩标签并排着让人看不明白。

到这一步,**球就到人手里了**:你去 review 这个 MR,合并,issue 自动就关了,进终态 `CLOSED`。顺的话,这条路就这么走完。

## 〔失败中间态——重点 · 约 2 分钟〕

下面是这次改得最多的地方,也是直接回应领导那条意见的:**「blocked 到底是调度器的毛病,还是 Agent 干活的毛病?用户得能分清,看一眼就知道去改哪儿。」**

〔指下面那个虚线框,里头俩红色状态〕以前咱们就一个笼统的 `blocked`,出了事还得扒日志才知道是谁的锅。这次我**按原因把它拆成了两个标签、两个状态**:

**头一个,`blocked-cc`,是 CC 侧的问题**,说白了就是 Claude Code 自己没干成——比如跑挂了、没产生改动、推送被拒了。这种情况它一般已经把半成品推到分支上了。**你一看到这个标签,就知道:去改 issue 的提示词、改 hulat 配置**,让它下回能干对,然后重跑就行。

**第二个,`blocked-dispatcher`,是调度器侧的问题**,就是编排或者环境出毛病了——准备阶段失败、子任务起不来、或者等不到回调卡住了。这种压根没产生有效产出。**你一看到它,就知道:这是基础设施的事,去改 dispatcher、修环境**,跟提示词一点关系没有。

这就是「看到状态就知道怎么办」落到实处:**你不用再翻什么 `block_reason` 了,标签名儿本身就告诉你该找哪边。**

〔指 TIMEOUT〕旁边还有个 `timeout`,这个我单独留成一个 CC 侧的状态,没拆。为啥不拆?因为我去翻了实现,`timeout` 只可能一种情况——就是 Claude Code 跑太久、超时了,**它天生就是 CC 侧的**。调度器那种「等不到回调」的超时,我归到刚才的 `blocked-dispatcher` 里了。所以 `timeout` 一个状态就够使,看到它就知道:任务太重了,去改提示词、把活儿拆小点,或者直接续跑。

## 〔FAILED 和关闭 · 约 40 秒〕

〔指 FAILED〕要是一个 issue 反复重试都不行、超过重试上限了,就提升到 `FAILED` 终态。这个状态**再也不会自动重跑了**,必须人来拍板:你可以照它上回是哪种 blocked,先把对应那边修了,再手动重新激活;实在不行就直接关掉。

再补一句,图上任何一个还没到终态的状态,人都能随时手动关进 `CLOSED`。为了线别太乱,我就画了 `FAILED → CLOSED` 这一条做个示意。

## 〔人怎么跟自动状态配合 · 约 1 分 30 秒〕

最后讲一个领导特别在意的:**人改完之后,自动模式下到底该「续着跑」还是「重头来」?**

〔指右下角那个紫框〕我把它做成了一套很明确的规矩,就两步:

**第一步,先看标签是哪边的锅,决定去改哪儿**——`blocked-cc` 和 `timeout` 去改提示词和 hulat;`blocked-dispatcher` 去改 dispatcher。

**第二步,决定重跑模式**,这个由人打的标签自动决定:
- 打的是 `continue`,而且只打这一个,就走 **continue 模式**——在上回的分支上接着干,上轮的产物留着,还会把上轮的总结、评审意见喂给它。**适合「方向是对的,就差点细节」。**
- 打的是 `retry`,就走 **fresh 模式**——回到干净基线从头来,上轮的东西归档、不带进来。**适合「上回跑偏了,或者规格改了」。**

一句话好记:**改改细节、接着做,就 `continue`;推倒重来,就 `retry`。** 这俩在图上、在系统里跑出来的结果是实打实不一样的,不是一回事。

## 〔收尾 · 约 30 秒〕

总结一下这一页的几个取舍,也正好对上大家上回提的意见:**起点改成标签了;两台状态机彻底分开、互不混画;不用人管的状态全沉下去了;失败态按原因拆开、做到自解释;`pr` 顶掉 `done` 让标签唯一;重跑的 `retry` 和 `continue` 也分清楚了。**

还有一条贯穿整张图、但画不进去的约定:**GitLab 上的实时标签才是状态的唯一真相**,磁盘上那些只是进度缓存,对不上的时候一律以 GitLab 为准。

这一页就讲到这儿,大家有意见我记下来。谢谢!

## 附:大概率被追问的几个问题（备着,不用念）

- **为什么 `timeout` 不也拆成两个?** —— 因为实现里 `timeout` 只由 acpx 跑超时产生,天生 CC 侧;调度器那种超时已经归到 `blocked-dispatcher` 了,再拆是多一个没必要的标签。
- **`pr` 顶掉 `done`,会不会丢信息?** —— 不会。`done` 只是建 MR 前的一个过渡,信息(commit、MR、Wiki)都在 MR 和总结里;`pr` 就代表「MR 已就绪待评审」,状态反而更唯一、更好认。
- **这套现在落地了吗?** —— 目前改的是状态机设计图。真要跑起来,还得配套改 `ensure_labels`、`set_issue_label`、`reconcile`、`campaign` 的分类和 executor 的打标签逻辑,这部分还没动。
- **拆出来的新标签谁来打?** —— 都是机器(子任务和调度器)根据失败归因自动打的;人只负责看标签、改对应那边、再打 `retry` / `continue`。`continue` 永远是人工动作,机器自己不会打。
